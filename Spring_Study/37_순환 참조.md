# 37_순환 참조

> 순환 참조(circular reference)

## 1. 개념

두 개 이상의 스프링 빈(Bean)이 서로를 의존(참조)하고 있어서, 서로가 서로를 생성되길 기다리다보니 영원히 생성되지 못하는 상태를 말한다. 순환 참조가 발생하면 결국 빈 등록에 실패하여 어플리케이션을 실행 못하게 된다.

`UnsatisfiedDependencyException` (의존성 불만족) 예외가 먼저 발생하고, `BeanCurrentlyInCreationException` (순환 참조) 예외를 확인할 수 있다.

### 1) 발생 원인

순환 참조의 발생 원인은 간단하게 말해서 A → B → A → ... 와 같은 관계가 계속해서 반복되는 것으로, 설계를 잘못하여 관계가 꼬여서 코드 간 참조 관계가 순환(무한 루프)되는 상황이다.

쉽게 말해서 A는 "B가 작업을 다 하면 내가 시작하겠다"라고 하고, B는 "A가 작업을 다 하면 내가 시작하겠다"라고 하는 상황인 셈이다.

### 2) 3단계 캐시 레이어

스프링이 빈을 만들 때, DefaultSingletonBeanRegistry는 내부적으로 3단계 캐시(Three-Level Cache)를 가진다.

※ DefaultSingletonBeanRegistry: 빈 팩토리(Bean Factory) 내부에 싱글톤을 보관하고 관리하는 저장소(Map형태로 보관한다. `Map<String, Object>`)

빈은 기본적으로 싱글톤이기 때문에 딱 한 번만 만든다. 따라서 이미 만든 객체가 있는지 확인하고, 있으면 꺼내주고 없으면 저장하는 로직으로 동작한다.

```java
// 1단계: singletonObjects        → 완전히 초기화된 Bean
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>();

// 2단계: earlySingletonObjects   → 껍데기 인스턴스 (필드 미주입 상태)
private final Map<String, Object> earlySingletonObjects = new HashMap<>();

// 3단계: singletonFactories      → Bean을 만들 수 있는 Factory
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>();
```

#### (1) singletonObjects (1차 캐시)

생성이 완전히 완료된(초기화된) 빈들의 저장소

#### (2) earlySingletonObjects (2차 캐시)

객체는 생성되었지만, 아직 의존성 주입이 되지 않은 미완성(껍데기) 빈들의 저장소

#### (3) singletonFactories (3차 캐시)

빈을 생성하는 팩토리 객체들의 저장소

## 2. Spring Boot 2.6 이전의 동작

### 1) 필드 주입

> "일단 빈 껍데기 인스턴스를 만들고 나중에 채운다"는 개념

Spring Boot 2.6 이전의 경우 field 주입(`@Autowired`)으로 의존성을 주입하는 경우

#### (1) A -> B -> A 순환 참조 시나리오

A의 필드에 B가 있고, B의 필드에 A가 있음.

- Service A를 만들다가 Service B가 필요하면, Service A를 미완성 상태(2nd 캐시)에 잠시 넣어둠.
- Service B를 만들다가 다시 Service A를 찾으면, 1차 캐시에는 없지만 2차 캐시에 있는 미완성 Service A를 일단 가져다 쓸 수 있음.
- 이 덕분에 무한 루프에 빠지지 않고 전체 빈 생성을 마칠 수 있게 됨.
  - 단, 생성자 주입은 객체 생성 자체가 안 되므로 이 메커니즘이 작동하지 않아 에러 발생

```
1. A 생성 시작
2. new A() 로 껍데기 인스턴스 생성
3. A의 ObjectFactory를 3단계(singletonFactories)에 등록
4. A의 필드 B를 채우려고 B 생성 시작
5. new B() 로 껍데기 인스턴스 생성
6. B의 필드 A를 채우려고 A를 요청
7. 1차 캐시 singletonObjects 확인 → 없음
8. 2차 캐시 earlySingletonObjects 확인 → 없음
9. 3차 캐시 singletonFactories 확인 → A의 Factory 있음!
10. Factory로 A의 early reference 꺼내서 2단계로 승격
11. B에 A의 껍데기 주입 완료 → B 초기화 완료 → 1단계로 승격
12. A에 B 주입 완료 → A 초기화 완료 → 1단계로 승격
```

<br>

### 2) 생성자 주입

생성자 주입은 Spring Boot 2.6 이전에서도 3단계 캐시로 해결이 불가능하다. (**Spring Framework도 마찬가지며, 이는 구조적으로 해결 불가능한 문제이다.**)

생성자 주입 방식의 경우`"인스턴스를 만들기 위해 의존성이 먼저 있어야"`하기 때문이다.

#### (1) A -> B -> A 순환 참조 시나리오

```
1. A 생성자 주입 시도
2. new A(B b) 호출해야 함
3. 그런데 B가 없으니 B 먼저 생성 시도
4. new B(A a) 호출해야 함
5. 이제는 A가 없으니 A 먼저 생성하려고 시도
6. A는 이미 "생성 중" 상태인데 인스턴스조차 없음!
7. BeanCurrentlyInCreationException 발생 ❌
```

<br>

## 3. Spring Boot 2.6 부터 변경

생성자 주입 방식의 경우 구조적으로 불가능하기 때문에 순환 참조를 허용하지 않는다. 버전에 따른 변경사항 없다. 그러나 필드 주입의 경우 Spring Boot 2.6부터 차단되었다.

### 1) 필드 주입

Spring Boot 2.6부터는 필드 주입 방식에서도 순환 참조 발생 시 기본적으로 차단되도록 변경되었다.

다음과 같은 옵션을 true로 활성화 해야만 가능하다.

```properties
spring.main.allow-circular-references=true
```

#### (1) 변경 이유?

> 순환 참조를 "해결한다"가 결국 올바른 코드 설계가 아니기 때문

즉, 애초에 설계 자체가 잘못된 것을 해결한다는 게 올바르지 않다고 봤기 때문이다.

만약에 이를 강제로 해결한다고 했을 경우,

- 3단계 캐시로 해결된 Bean은 "완전히 초기화되지 않은 상태의 참조"를 하고 있음
- 스프링 AOP Proxy, @Transactional 같은 후처리가 적용되기 전 객체가 주입될 위험이 있음
- 이렇게 될 경우 실제로는 Proxy 객체를 통해서 동작해야 하는 로직이 원본 객체를 참조하게 되어 알 수 없는 버그를 마주하게 될 수도 있다. (디버깅 또한 매우 어려움)

대표적으로  @Transactional이 걸린 Bean의 경우, Spring은 트랜잭션 처리를 위해 실제 Bean 대신 프록시 객체를 만들어서 주입한다. 그런데 순환 참조 상황에서 껍데기 early reference가 먼저 다른 Bean에 주입되면 이는 프록시 객체가 아닌 원본 객체일 수 있다. 결과적으로 트랜잭션이 안 걸리는 버그가 생기는데, 재현 및 원인 분석이 굉장히 힘든 케이스가 발생한다. 그래서 아예 기본 차단으로 바꾼 것이다.

### 2) Spring Framework

Spring Framework는 `allowCircularReferences = true`로  필드 주입 순환 참조를 기본값으로 허용하고 있다.

## 4. (참고) 스프링의 의존성 주입 방식

> 의존성 주입 방식(DI; Dependecy Injection)

스프링 컨테이너가 빈을 생성하고 의존  관계를 맺어주는 과정.

### 1) 필드 주입

필드 변수에 직접 `@Autowired`어노테이션을 붙이는 방식

```java
@Service
public class TicketService {
    @Autowired
    private TicketRepository repository; // 필드 주입
}
```

<br>

### 2) 생성자 주입

생성자를 통해 의존성을 주입하는 방식으로, 권장되는 방식이다. 직접 생성자를 통해서 매개변수(인자)를 통해 주입하거나 `private final`과 롬복의 `@RequiredArgsConstructor` 어노테이션을 통해 주입하는 방식이 있다.

```java
// 기본
@Service
public class TicketService {
    private final TicketRepository repository;

    public TicketService(TicketRepository repository) {
        this.repository = repository;
    }
}

// @RequiredArgsConstructor
@Service
@RequiredArgsConstructor
public class OrderService {
    private final PaymentService paymentService;
}
```

<br>

### 3) 수정자 주입

setter 메서드를 통해 주입받는 방식으로 실제로는 거의 사용되지 않는다.
