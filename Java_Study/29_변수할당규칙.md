# 29_변수할당규칙

> Java의 인스턴스 변수, 로컬 변수 할당 규칙

Java에서 변수는 선언 위치에 따라 완전히 다른 초기화 규칙을 가진다.

가장 큰 차이는 "초기값을 보장하는가/하지 않는가"이다.

## 1. 인스턴스 변수, 필드 변수

클래스 내부(메서드 바깥)에 선언된 변수.

이들 변수는 JVM이 객체를 힙에 생성할 때, 자동으로 기본값을 할당한다.

```java
public class Example {
    int num;           // 0
    boolean flag;      // false
    String name;       // null
    double price;      // 0.0

    void print() {
        System.out.println(num);   // 0
        System.out.println(name);  // null
    }
}
```

이 때, 기본형(Primitive)의 경우 해당 기본형 타입이 기본적으로 가질 수 있는 초기값 (0, 0.0, false 등)으로 초기화되며, 참조형(Reference) 타입의 경우 null로 초기화된다.

### 1) 원리

이것이 가능한 이유는 객체가 힙에 할당될 때, 그 메모리 영역이 0으로 클리어(zero-out)되기 때문이다.

#### (1) 힙 메모리와 zero-out

`new` 키워드로 객체를 생성하면 JVM은 힙(heap) 영역에 필요한 크기만큼 메모리를 확보한다. (힙 메모리)

이때 확보한 메모리 영역의 모든 바이트를 0으로 채우는 작업을 zero-out (또는 zero-initialization)이라고 한다.

```java
class User {
    int age;
    String name;
    boolean active;
}

User u = new User();  // 이 시점에 메모리에서 일어나는 일:
                       // 1. 힙에서 User 객체 크기만큼 공간 할당
                       // 2. 그 공간을 전부 0x00 으로 채움 (zero-out)
                       // 3. 객체 헤더 세팅
                       // 4. 명시적 초기화/생성자 실행
```

즉, 기본값은 별도로 JVM이 직접적으로 할당(부여)하는 것이 아니라 0으로 채워진 메모리를 각 타입으로 해석한 결과를 의미하는 것이다.

이는 JVM이 보장하는 의무적인 스펙이며, 결과적으로 개발자가 관찰하는 시점에는 항상 zero-out되어 기본값으로 표현된다. (내부 구현에 따라 zero-out 시점에 차이는 있음)

| 타입             | 메모리 (0으로 채워진 상태) | 해석된 값  |
| ---------------- | -------------------------- | ---------- |
| `int` (4 bytes)  | `00 00 00 00`              | `0`        |
| `long` (8 bytes) | `00 00 00 00 00 00 00 00`  | `0L`       |
| `double`         | `00 00 00 00 00 00 00 00`  | `0.0`      |
| `boolean`        | `00`                       | `false`    |
| `char`           | `00 00`                    | `'\u0000'` |
| 참조형           | `00 00 00 00 00 00 00 00`  | `null`     |

#### (2) static 변수

static 변수의 경우 힙이 아니라 메서드 영역(Method Area)에 저장되지만, 초기화 보장은 동일하다.

## 2. 로컬 변수

메서드, 생성자, 블록 내부에서 선언된 변수.

JVM이 기본값을 주지 않는 변수로, 개발자가 직접 초기화해야 한다.

```java
void method() {
    int x;
    System.out.println(x);  // 컴파일 에러 "variable x might not have been initialized"
}
```

### 1) 확정 할당(Definite Assignment) 규칙

"로컬 변수를 읽는 시점에, 컴파일러가 분석 가능한 모든 실행 경로에서 그 변수가 반드시 할당되어 있음이 증명되어야 한다."

이는 "선언 시점에 초기화해라"가 아니라, "사용하기 전까지 모든 경로에서 할당이 보장되어야 한다"를 의미한다.

#### (1) 예시

- 케이스1; 선언 후 사용 전 할당 (가능)

```java
int x;
x = 10;
System.out.println(x); // 10 (가능)
```

- 케이스2; 선언 후 한쪽 분기에서만 할당 (불가능)

```java
int x;
if (cond) {
    x = 10;
}
System.out.println(x);  // 컴파일 에러 - cond가 false인 조건에서는 x가 할당되지 않아 확정 할당 규칙에 위배
```

- 케이스3; 모든 분기에서 할당 (가능)

```java
int x;
if (cond) {
    x = 10;
} else {
    x = 20;
}
System.out.println(x); // 가능 (모든 분기에서 할당되어 있음 - 10 아니면 20임)
```

- 케이스4; 한쪽 분기가 종료된 경우 (가능)

```java
int x;
if (cond) {
    x = 10;
} else {
    return;  // 또는 throw
}
System.out.println(x); // else로 가면 return되어 x가 사용되지 않음. if로 갔으면 x를 사용하기 때문에 가능
```

- 케이스5; 상수 조건 (컴파일러가 인식, 가능)
  - 단, 컴파일러는 런타임 값을 추론하지 않는다. 정적으로 분석 가능한 흐름만 고려한다. 
  - 즉 케이스2에서 cond 값이 항상 true이더라도, 양쪽 경로에서 변수에 대한 할당이 보장되지 않으면 컴파일 에러가 발생한다.

```java
int x;
if (true) {
    x = 10;
}
System.out.println(x); // 10 (가능)
```

### 2) 로컬 변수가 초기화 혜택을 받지 못하는 이유

로컬 변수는 JVM의 스택 영역에 저장된다. 스택의 경우 메서드 호출마다 프레임이 push/pop되는 매우 빈번한 영역이다. (변수의 수명 주기가 짧음)

만약에 매 호출마다 메모리를 zero-out한다면, 성능상으로는 메서드 호출에 따른 오버헤드가 누적되어 좋지 못하고, 어차피 컴파일러가 정적 분석을 통해 미초기화된 변수 사용을 확인할 수 있는데 불필요한 작업을 하는 셈이다.

## 3. 설계 의도

- 인스턴스 변수
  - 객체의 경우 수명이 길고, 객체의 변수는 여러 메서드에서 공유된다.
  - 이 경우 누가 언제 첫 할당을 할지 정적 분석이 사실상 불가능하다. 즉, A메서드가 먼저 호출될지, B 메서드가 먼저 호출될지는 정적 분석으로 판단이 불가하기 때문에 자동 초기화가 합리적인 셈이다.
- 로컬 변수
  - 단일 메서드, 코드 블럭 안에서만 살아 있기 때문에 수명이 짧다.
  - 컴파일러가 정적 분석을 통해서 완전히 추적 가능하다. 따라서 초기화되지 않은 사용을 컴파일 타임에 잡아내는 게 가능하며, 더 안전하다. (사전에 버그를 미리 차단)

이러한 특징 때문에 Java 설계자는 힙은 런타임이, 스택은 컴파일러가 안전성을 책임진다는 분업 구조를 채택한 것이다.

- 힙은 한 번 할당하면 객체 수명이 길고 여러 메서드/스레드에 걸쳐 공유되기 때문에 언제 사용될지 몰라 정적 분석이 불가능하다. => 런타임 시 zero-out으로 초기화하여 해결.
- 스택은 단일 메서드 스코프라 정적 분석이 가능하다. => 확정 할당 규칙으로 해결
